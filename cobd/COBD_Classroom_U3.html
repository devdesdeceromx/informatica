Sección 2 · Teoría

Transacciones y propiedades ACID

¿Qué es una transacción?

Una transacción es una secuencia de operaciones SQL que se ejecutan como una sola unidad indivisible. O todas se completan exitosamente (COMMIT), o ninguna tiene efecto (ROLLBACK). No existe término medio.

Ejemplo clásico: transferencia bancaria. Se descuenta de cuenta A y se abona en cuenta B. Si el sistema falla entre medio, el banco no puede quedarse con el dinero en el aire — ambas operaciones deben ocurrir juntas o ninguna.

A
Atomicidad
Todo o nada. Si una operación falla, todas las anteriores se deshacen con ROLLBACK.
C
Consistencia
La BD pasa de un estado válido a otro. Las restricciones de integridad siempre se cumplen.
I
Aislamiento
Las transacciones concurrentes no se "ven" entre sí hasta que hacen COMMIT.
D
Durabilidad
Una vez con COMMIT, los cambios permanecen aunque el servidor se reinicie o falle.

Sintaxis básica en MySQL

MySQL · transaccion_transferencia.sql
-- Estructura básica de una transacción
START TRANSACTION;  -- o BEGIN;

  -- Operaciones que forman la unidad
  UPDATE cuentas SET saldo = saldo - 1000 WHERE id = 1;
  UPDATE cuentas SET saldo = saldo + 1000 WHERE id = 2;

COMMIT;  -- Confirmar: los cambios son permanentes

-- Si algo falla, se usa:
ROLLBACK;  -- Deshacer: ningún cambio se aplica
Punto clave para la actividad

En MySQL, las transacciones solo funcionan con tablas de tipo InnoDB (el motor por defecto). Las tablas MyISAM no soportan transacciones — hacen los cambios inmediatamente y no hay rollback posible.

SAVEPOINT — puntos de control intermedios

MySQL · savepoints.sql
START TRANSACTION;
  INSERT INTO pedidos VALUES (1, 'Laptop', 15000);

  SAVEPOINT sp_pedido;  -- punto de guardado parcial

  INSERT INTO detalles VALUES (1, 'Intel i7');
  -- Si este INSERT falla:
  ROLLBACK TO SAVEPOINT sp_pedido;  -- vuelve solo hasta aquí
  -- El pedido se mantiene, solo el detalle se cancela

COMMIT;
Sección 3 · Teoría

Protocolo de bloqueo en dos fases (2PL)

¿Por qué necesitamos bloqueos?

Cuando dos usuarios modifican el mismo registro al mismo tiempo, los cambios de uno pueden sobrescribir los del otro. Los bloqueos (locks) controlan el acceso concurrente garantizando que solo un usuario a la vez modifique un dato.

Tipo de bloqueo
🔍 Bloqueo Compartido (S)
Shared Lock — permite que múltiples transacciones lean el dato simultáneamente. Ninguna puede modificarlo mientras otro lo lee. Se pide al hacer SELECT ... FOR SHARE.
Tipo de bloqueo
✏️ Bloqueo Exclusivo (X)
Exclusive Lock — solo una transacción puede tener acceso. Ninguna otra puede leer ni escribir. Se pide al hacer SELECT ... FOR UPDATE o al ejecutar un UPDATE/DELETE.

La regla fundamental del 2PL

Una transacción nunca puede adquirir un nuevo bloqueo después de haber liberado uno. Esta regla garantiza la seriabilidad — el resultado de transacciones concurrentes es equivalente a ejecutarlas una por una en algún orden.

MySQL · bloqueo_selectivo.sql
-- Bloqueo compartido: múltiples pueden leer
SELECT * FROM productos
WHERE id = 10
FOR SHARE;  -- nadie puede modificar este registro hasta que hagamos COMMIT

-- Bloqueo exclusivo: solo yo puedo leer y escribir
SELECT * FROM productos
WHERE id = 10
FOR UPDATE;  -- nadie más puede ni siquiera leerlo

-- Bloqueo a nivel de tabla completa
LOCK TABLES ventas WRITE;
  -- ... operaciones ...
UNLOCK TABLES;

Concurrencia, problemas y Deadlock

Problemas de la concurrencia sin control

Problema¿Qué ocurre?Solución
Lectura sucia
Dirty Read		T2 lee datos que T1 modificó pero aún no confirmó. Si T1 hace ROLLBACK, T2 leyó algo que nunca existió.Nivel de aislamiento READ COMMITTED
Lectura no repetible
Non-repeatable Read		T1 lee un valor, T2 lo modifica y confirma, T1 vuelve a leerlo y obtiene un valor diferente.Nivel REPEATABLE READ (default en MySQL)
Lectura fantasma
Phantom Read		T1 consulta con WHERE y obtiene N filas. T2 inserta una fila que cumple el WHERE. T1 vuelve a consultar y ahora obtiene N+1.Nivel SERIALIZABLE
Actualización perdida
Lost Update		T1 y T2 leen el mismo valor, ambos calculan y escriben — el resultado de T1 queda sobreescrito por T2.Bloqueos exclusivos con FOR UPDATE

Deadlock (Abrazo mortal)

Ocurre cuando dos transacciones se bloquean mutuamente esperando recursos que la otra tiene:

T1 tiene
Lock(A)
espera Lock(B)
⟷
T2 tiene
Lock(B)
espera Lock(A)
→
💀 Deadlock
ninguna avanza

MySQL lo detecta automáticamente y mata a la transacción víctima (generalmente la más reciente). Esa transacción recibe un error y debe ser reiniciada por la aplicación.

MySQL · niveles_aislamiento.sql
-- Ver el nivel de aislamiento actual
SELECT @@transaction_isolation;
-- Resultado default MySQL: REPEATABLE-READ

-- Cambiar para la sesión actual
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;
SET SESSION TRANSACTION ISOLATION LEVEL SERIALIZABLE;

-- Ver estado de bloqueos en tiempo real
SHOW ENGINE INNODB STATUS;
Técnica para prevenir deadlocks

Siempre adquiere los bloqueos en el mismo orden en todas las transacciones. Si T1 siempre bloquea A antes que B, y T2 también, nunca habrá deadlock.

Sección 5 · Ejemplo guiado

Caso práctico · Sistema de inventario

Lee el código antes de ejecutarlo

Este ejemplo simula una venta en el sistema de la empresa. Analiza cada paso, luego cópialo al simulador de la Sección 11 para verlo funcionar.

Escenario: Procesar una venta

Cuando se vende un producto: (1) se registra la orden, (2) se descuenta el stock, (3) se actualiza el total. Si algo falla en cualquier paso, todo debe revertirse — no queremos órdenes sin stock descontado.

MySQL · venta_transaccion.sql — Ejemplo completo con manejo de errores
-- Iniciamos la transacción
START TRANSACTION;

-- Paso 1: Bloquear el producto para modificarlo
SELECT id, nombre, stock, precio
FROM products
WHERE id = 1
FOR UPDATE;  -- nadie más puede modificar este producto

-- Paso 2: Verificar stock suficiente (esto lo hace la app)
-- Si stock < cantidad_pedida → ROLLBACK

-- Paso 3: Descontar stock
UPDATE products
SET stock = stock - 2
WHERE id = 1 AND stock >= 2;

-- Paso 4: Registrar la orden
INSERT INTO orders (employee_id, order_date, total)
VALUES (1, NOW(), 2 * (SELECT precio FROM products WHERE id = 1));

-- Paso 5: Todo salió bien → confirmar
COMMIT;

-- Si algo falló → deshacer todo
-- ROLLBACK;
¿Por qué el AND stock >= 2 en el UPDATE?

Es una verificación atómica: si el UPDATE afecta 0 filas, sabemos que el stock era insuficiente. Esto es más seguro que un SELECT separado porque entre el SELECT y el UPDATE otro usuario podría haber comprado el último producto.

Sección 6 · Teoría

Control de acceso · Usuarios, roles y privilegios

¿Por qué controlar el acceso?

En una empresa real, no todos deben poder hacer todo en la base de datos. Un empleado de ventas puede consultar productos pero no puede modificar sueldos. Un DBA puede hacer todo pero no acceder a datos personales sin autorización. El principio de mínimo privilegio dice: otorgar solo lo necesario para el trabajo.

Jerarquía: Usuario → Rol → Privilegios

Nivel 1
👤 Usuario
Identidad con contraseña. Se crea con CREATE USER. Por sí solo no puede hacer nada hasta que se le otorguen privilegios.
Nivel 2
🏷️ Rol
Conjunto de privilegios agrupados bajo un nombre. Facilita la gestión: en lugar de dar 10 permisos individualmente, asignas un rol.
Nivel 3
🔑 Privilegios
SELECT, INSERT, UPDATE, DELETE, CREATE, DROP, GRANT OPTION, ALL PRIVILEGES. Pueden ser globales, por BD o por tabla.
MySQL · usuarios_y_roles.sql
-- ① Crear usuarios
CREATE USER 'vendedor1'@'localhost' IDENTIFIED BY 'Pass123!';
CREATE USER 'analista1'@'%' IDENTIFIED BY 'Anal2024!';

-- ② Crear roles
CREATE ROLE 'rol_ventas', 'rol_reportes', 'rol_admin';

-- ③ Asignar privilegios al rol
GRANT SELECT, INSERT ON empresa_cobd.orders TO 'rol_ventas';
GRANT SELECT ON empresa_cobd.* TO 'rol_reportes';
GRANT ALL PRIVILEGES ON empresa_cobd.* TO 'rol_admin';

-- ④ Asignar rol a usuario
GRANT 'rol_ventas' TO 'vendedor1'@'localhost';
GRANT 'rol_reportes' TO 'analista1'@'%';

-- ⑤ Verificar privilegios
SHOW GRANTS FOR 'vendedor1'@'localhost';

-- ⑥ Revocar privilegios
REVOKE INSERT ON empresa_cobd.orders FROM 'rol_ventas';

-- ⑦ Eliminar usuario
DROP USER 'vendedor1'@'localhost';
Sección 7 · Teoría

Cifrado de datos y estrategias de respaldo

Cifrado en MySQL
🔐 AES_ENCRYPT
MySQL provee funciones nativas para cifrar columnas sensibles como contraseñas, números de tarjeta o CURP. Solo quien tiene la clave puede leer el dato.
Hashing seguro
🔏 SHA2 / bcrypt
Para contraseñas, nunca se guarda el valor original — solo su hash. SHA2() es unidireccional: de él no puedes recuperar la contraseña original.
MySQL · cifrado_datos.sql
-- Cifrar con AES (clave simétrica)
INSERT INTO employees (name, salary, curp_cifrada)
VALUES (
  'Ana García',
  15000,
  AES_ENCRYPT('GAAA990101HTCRNB01', 'mi_clave_secreta')
);

-- Descifrar para leer
SELECT name,
  AES_DECRYPT(curp_cifrada, 'mi_clave_secreta') AS curp_legible
FROM employees;

-- Hashear contraseñas (unidireccional)
INSERT INTO users (username, password_hash)
VALUES ('ana', SHA2('MiPassword123', 256));

-- Verificar contraseña (comparar hash)
SELECT * FROM users
WHERE username = 'ana'
  AND password_hash = SHA2('MiPassword123', 256);

Estrategia recomendada en producción

Full backup los domingos + Incrementales de lunes a sábado. Así en el peor caso solo pierdes los datos del día de hoy. Los respaldos deben almacenarse en un servidor diferente o en la nube.

Shell · mysqldump — respaldo desde línea de comandos
-- Respaldo completo de una base de datos
-- (ejecutar en terminal, no en MySQL)
mysqldump -u root -p empresa_cobd > empresa_20260623.sql

-- Respaldo de tabla específica
mysqldump -u root -p empresa_cobd employees > employees_bak.sql

-- Restaurar desde respaldo
mysql -u root -p empresa_cobd < empresa_20260623.sql
Sección 8 · Ejemplo guiado

Caso práctico · Configurar seguridad completa

Escenario real

La empresa tiene tres tipos de empleados que usan la base de datos: el DBA (todo), el gerente de ventas (consulta + inserta órdenes) y el analista financiero (solo consulta salarios y ventas). Configura los accesos correctos.

MySQL · seguridad_completa.sql — solución paso a paso
-- ══ PASO 1: Crear roles según función ══
CREATE ROLE 'dba_empresa';
CREATE ROLE 'gerente_ventas';
CREATE ROLE 'analista_financiero';

-- ══ PASO 2: Definir qué puede hacer cada rol ══
GRANT ALL PRIVILEGES ON empresa_cobd.* TO 'dba_empresa';

GRANT SELECT ON empresa_cobd.products TO 'gerente_ventas';
GRANT SELECT, INSERT ON empresa_cobd.orders TO 'gerente_ventas';

GRANT SELECT(name, salary, department_id)
  ON empresa_cobd.employees TO 'analista_financiero';
GRANT SELECT ON empresa_cobd.orders TO 'analista_financiero';

-- ══ PASO 3: Crear usuarios reales ══
CREATE USER 'carlos_dba'@'localhost' IDENTIFIED BY 'Dba$2024!';
CREATE USER 'laura_ventas'@'%' IDENTIFIED BY 'Vtas!456';
CREATE USER 'pedro_analista'@'%' IDENTIFIED BY 'Ana#789';

-- ══ PASO 4: Asignar roles ══
GRANT 'dba_empresa' TO 'carlos_dba'@'localhost';
GRANT 'gerente_ventas' TO 'laura_ventas'@'%';
GRANT 'analista_financiero' TO 'pedro_analista'@'%';

-- ══ PASO 5: Aplicar cambios ══
FLUSH PRIVILEGES;
